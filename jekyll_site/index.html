TownSuite.WorkQueues is a lightweight message bus and work queue for .NET, backed by SQL Server, PostgreSQL, Redis, or SQLite. No broker to deploy, no new infrastructure to babysit — your queue lives where your data already does.

01

Why it exists

Most systems don't need Kafka. They need a table and a transaction.

0 brokers

No new infrastructure

The queue is a table in the database you already run, back up, and monitor. Nothing extra to deploy, patch, or page anyone about at 3 a.m.

BEGIN…COMMIT

Transactional by design

Dequeue happens inside your own DbTransaction. A message leaves the queue only when your work commits — process and acknowledge atomically.

≤ 10k calls/s

Honest about scale

Built for low-load systems and says so up front. Benchmarks are published, limits are documented, and the README tells you when to use something else.

02

The API

Publish and subscribe. A consumer per message type.

OrderConsumer.csconsumer
public class OrderConsumer
    : IConsumer<OrderSubmitted>
{
    public async Task Consume(
        ConsumeContext<OrderSubmitted> ctx)
    {
        // ctx.Message    — your typed payload
        // ctx.MessageId  — stable UUID per message
        // ctx.SentTime   — when Publish() was called
        await Process(ctx.Message);
        // throw to retry; no action = acknowledged
    }
}
Program.cswire-up & publish
var bus = new PostgresMessageBus(options, logger);

// singleton consumer instance
bus.Subscribe(new OrderConsumer());

// or DI-scoped — resolved per message
bus.Subscribe<OrderSubmitted, OrderConsumer>();

// publish — channel derived from type name
await bus.Publish(new OrderSubmitted
{
    OrderId  = Guid.NewGuid(),
    Customer = "alice@example.com"
});

Multiple consumers can subscribe to the same message type — each gets a copy. For direct database access without the polling loop, use IWorkQueue.Enqueue / Dequeue inside your own DbTransaction.

03

Reliability

Durable delivery, without a sidecar.

MaxRetries = 3

Automatic retries

A consumer that throws is retried up to MaxRetries times. Configure RetryDelay to back off between attempts. After the final attempt the row is dead-lettered — polling skips it permanently.

ReplayDeadLettered<T>

Dead-letter & replay

Dead-lettered rows stay in the table for inspection. Call ReplayDeadLettered<T>() or run a single UPDATE to reset them for redelivery. No separate topic or stream to manage.

IConsumer<Fault<T>>

Failure notifications

Subscribe a fault consumer and it fires whenever a message is dead-lettered, carrying the original payload, exception type, stack trace, attempt count, and timestamp.

Program.csscheduled delivery
// deliver no earlier than 5 minutes from now
await bus.Publish(
    new ReminderEmail { UserId = userId },
    deliverAfter: TimeSpan.FromMinutes(5));

// back off 30 s between retry attempts
options.RetryDelay = TimeSpan.FromSeconds(30);

// replay all dead-lettered messages of this type
int replayed = await bus
    .ReplayDeadLettered<OrderSubmitted>();
AlertConsumer.csfault hook
bus.SubscribeFault<OrderSubmitted>(
    new OrderAlertConsumer());

class OrderAlertConsumer
    : IConsumer<Fault<OrderSubmitted>>
{
    public Task Consume(
        ConsumeContext<Fault<OrderSubmitted>> ctx)
    {
        var f = ctx.Message;
        // f.OriginalMessage  f.ExceptionType
        // f.AttemptCount     f.FaultedAt
        return Notify(f);
    }
}
04

Backends

Bring whichever store you already trust.

[x]SQL Server

One table and two stored procedures. Run the setup scripts and the queue lives in dbo alongside your data.

dbo.WorkQueue.sql
dbo.WorkQueue_Enqueue.sql

[x]PostgreSQL

Same shape, same API. The fastest backend in published benchmarks — burstable to roughly 19,000 calls per second.

public.WorkQueue.sql
public.WorkQueue_Enqueue.sql
public.WorkQueue_Dequeue.sql
[x]Redis

For when the queue can be in memory and the database round-trip is overhead you don't want. No setup scripts required.

no scripts — point it at
your Redis connection
[~]SQLite local dev

Share a single .db file across processes on the same machine — a web frontend enqueuing, a worker consuming — with no database server to run. Uses a visibility-timeout claim pattern with WAL mode for concurrent access. Not intended for production.

06

Know its limits

Use it when it fits. Skip it when it doesn't.

A good fit

  • Background jobs in line-of-business apps — exports, reports, notifications
  • Sustained throughput under ~10,000 calls per second
  • Work that must commit atomically with your data
  • Teams who'd rather not run a message broker

Reach for Kafka or RabbitMQ instead

  • Sustained high-load or firehose event streams
  • Fan-out pub/sub to many independent subscribers
  • Push delivery — consumers here poll the table
  • Cross-system streaming pipelines and replay
